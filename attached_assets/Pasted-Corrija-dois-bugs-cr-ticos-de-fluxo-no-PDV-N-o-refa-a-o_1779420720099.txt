PROBLEMA 1 — Pedido já pago volta a mostrar botão "Pagar"

Fluxo onde deu bug:
1. Criei um pedido de balcão manual.
2. Enviei para cozinha.
3. Fui em Pedidos e cliquei em Pagar.
4. O pagamento foi registrado corretamente.
5. Na tela Pedidos apareceu "Dar baixa", correto.
6. Depois fui na Cozinha e marquei o pedido como pronto.
7. Voltei para Pedidos.
8. O pedido voltou a mostrar "Pagar", mesmo já estando pago.

Isso está errado.

REGRA ABSOLUTA:
Se o pedido já tem payment registrado, nunca deve aparecer botão "Pagar" de novo.

O botão "Pagar" só pode aparecer se:
- não existe payment registrado para esse pedido
- o pedido não está closed/finalizado
- o pedido não está cancelled
- o pedido não é delivery com paymentTiming = on_delivery aguardando baixa no Caixa

Se já existe payment:
- mostrar "Dar baixa" quando ainda faltar encerramento operacional
- ou mostrar "Finalizado" quando já estiver closed
- mas nunca mostrar "Pagar" novamente

==================================================
1. CORRIGIR BACKEND / API DE PEDIDOS
==================================================

Verifique se a API de pedidos retorna informação suficiente para o frontend saber se o pedido já foi pago.

Cada pedido deve retornar algo como:
- paid
- hasPayment
- paidAt
- paymentStatus
- paymentsCount
- ou qualquer campo equivalente já existente

Se não existir, adicionar de forma segura.

A tela Pedidos não deve decidir apenas pelo order.status.
Ela precisa considerar se existe pagamento.

IMPORTANTE:
orders.status pode mudar para ready quando cozinha marca pronto.
Mas isso não pode apagar/ignorar o fato de que o pedido já foi pago.

==================================================
2. CORRIGIR AÇÃO DA COZINHA AO MARCAR PRONTO
==================================================

Quando a cozinha marca um pedido como pronto:

Se o pedido NÃO foi pago:
- status pode virar ready
- botão Pagar pode aparecer, se o tipo permitir

Se o pedido JÁ foi pago:
- não voltar para estado que mostre Pagar
- manter payment existente
- deixar pronto para "Dar baixa" ou "Finalizar"
- se for balcão/takeaway, mostrar "Dar baixa"
- se for mesa, mostrar "Dar baixa" ou finalizar conforme regra de mesa
- se for delivery pago antes, seguir fluxo de rota/entrega

A cozinha não pode desfazer estado financeiro.

==================================================
3. CORRIGIR FRONTEND DA TELA PEDIDOS
==================================================

Na tela Pedidos, reescrever a lógica dos botões com prioridade correta:

Prioridade:

1. Se status = cancelled:
   - mostrar Cancelado
   - não mostrar Pagar

2. Se pedido tem payment registrado:
   - não mostrar Pagar
   - se status ainda não é closed:
      mostrar Dar baixa, quando operacionalmente permitido
   - se status é closed:
      mostrar Finalizado

3. Se delivery paymentTiming = on_delivery:
   - não mostrar Pagar antes da baixa financeira
   - se deliveryStatus = awaiting_settlement:
      mostrar "Aguardando baixa financeira"
      orientar que a baixa é no Caixa
   - não mostrar Pagar

4. Se não tem payment:
   - mostrar Pagar apenas para tipos e estados onde pagamento no caixa faz sentido:
      balcão, mesa, takeaway, delivery paymentTiming=now

5. Se cozinha marcou pronto em pedido já pago:
   - continuar mostrando Dar baixa
   - nunca voltar para Pagar
