OBJETIVO:
Corrigir o buraco crítico do sistema: pedidos delivery com paymentTiming = "on_delivery" não podem ser finalizados/fechados sem antes dar baixa financeira no caixa.

Hoje o problema é:
- motoboy conclui a rota
- pedido é considerado entregue/fechado
- mas nenhum payment é criado
- nenhum cash_movement é criado
- o caixa fecha errado
- o dashboard pode contar errado

Quero corrigir isso com o fluxo correto.

==================================================
PARTE 1 — ADICIONAR ESTADO "AGUARDANDO BAIXA"
==================================================

Adicionar suporte ao status:

deliveryStatus = "awaiting_settlement"

Esse status significa:
- entrega foi concluída fisicamente
- o motoboy voltou ou precisa prestar conta
- o pagamento foi combinado para ser feito na entrega
- ainda falta registrar o recebimento no caixa

Não finalizar financeiramente o pedido nesse estado.

==================================================
PARTE 2 — AJUSTAR CONCLUSÃO DE ROTA
==================================================

Alterar o endpoint/ação de concluir rota.

Quando uma rota for concluída:

Para cada pedido da rota:

1. Se paymentTiming = "now":
   - significa que o pedido já foi pago antes
   - pode marcar:
     - deliveryStatus = delivered
     - status = closed
     - closedAt = now, se existir

2. Se paymentTiming = "on_delivery":
   - NÃO criar payment automaticamente
   - NÃO criar cash_movement automaticamente
   - NÃO marcar como closed ainda
   - marcar:
     - deliveryStatus = awaiting_settlement
     - status deve permanecer ativo ou usar status apropriado para aguardando baixa
   - esse pedido deve aparecer no Caixa como pendente de baixa

Regra importante:
Pedido pago na entrega só fecha depois da baixa financeira no caixa.

==================================================
PARTE 3 — CORRIGIR ENTREGA INDIVIDUAL
==================================================

Verificar também endpoint/ação tipo:

POST /delivery/orders/:id/delivered

ou equivalente.

Se o pedido tiver paymentTiming = "on_delivery":
- não pode fechar o pedido direto
- deve marcar deliveryStatus = awaiting_settlement
- deve aguardar baixa financeira

Se paymentTiming = "now":
- pode marcar delivered/closed normalmente, pois já foi pago

==================================================
PARTE 5 — CRIAR ENDPOINT DE BAIXA FINANCEIRA
==================================================

Criar endpoint:

POST /delivery/orders/:id/settle

Esse endpoint registra a baixa financeira do pedido entregue.

Payload sugerido:
{
  "method": "cash" | "pix" | "credit" | "debit" | "voucher",
  "amountReceived": number,
  "notes": string optional
}

Regras:

1. Precisa existir caixa aberto.
Se não houver caixa aberto:
- retornar erro claro:
"Abra o caixa antes de registrar recebimentos de entrega."

2. Evitar duplicidade.
Se o pedido já tiver payment registrado:
- retornar 409:
"Pagamento deste pedido já foi registrado."

3. Validar status.
Só permitir baixa se:
- deliveryStatus = awaiting_settlement
- paymentTiming = on_delivery

4. Se método for dinheiro:
- validar amountReceived >= total do pedido
- calcular troco real

5. Criar payment.
6. Criar cash_movement do tipo payment no caixa aberto.
7. Atualizar pedido:
- deliveryStatus = delivered
- status = closed
- paidAt = now, se o campo existir
- closedAt = now, se o campo existir

8. Fazer tudo dentro de transação.
Se qualquer etapa falhar, nada deve ficar salvo pela metade.
